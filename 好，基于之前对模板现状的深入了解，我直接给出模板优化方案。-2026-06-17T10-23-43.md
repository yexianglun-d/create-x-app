## 模板优化方案

### 当前问题诊断

| 问题 | 具体表现 |
|------|---------|
| **模板是 Demo 不是脚手架** | `react-vite-ts` 有完整的 CRM 客户管理系统，`java-fullstack` 有硬编码的 `seedDefaults()` 业务数据 |
| **模板间差异模糊** | `react-vite-ts` 和 `react-admin` 都是 React + Tailwind，业务逻辑雷同 |
| **去模板感矫枉过正** | 从"欢迎使用 create-x-app"变成了"华东渠道续约"等具体业务，反而更难复用 |
| **架构教学缺失** | 用户拿到代码不知道为什么要这样组织，没有架构说明 |

---

### 优化方向

**核心原则：模板 = 干净的工程起点 + 最佳实践配置 + 架构示例（可选）**

#### 1. 每套模板分两层：`minimal`（默认）和 `example`（可选）

```
react-vite-ts/
├── minimal/          # 默认：干净脚手架
│   ├── src/
│   │   ├── App.tsx   # 只有一个 Hello World + 路由示例
│   │   ├── components/
│   │   │   └── Layout.tsx   # 布局壳（Header/Sidebar/Content）
│   │   ├── pages/
│   │   │   ├── Home.tsx     # 空白首页
│   │   │   └── About.tsx    # 空白示例页
│   │   └── hooks/           # 空目录，提示用户放自定义 hooks
│   ├── vite.config.ts
│   ├── tsconfig.json
│   ├── eslint.config.js      # 已配好
│   ├── .prettierrc           # 已配好
│   └── package.json          # 只有必要依赖，无业务依赖
│
└── example/          # 可选：带架构示例
    └── (带一个 TodoMVC 或计数器示例，展示状态管理/路由/API 调用)
```

**CLI 交互**：
```
? 选择模板模式：
  ● 最小起步（推荐）— 干净工程结构，适合直接开发
  ○ 架构示例 — 包含示例代码，适合学习架构
```

#### 2. 模板定位要正交，不能重叠

| 模板 | 定位 | 核心价值 |
|------|------|---------|
| `react-vite-ts` | **SPA 起步** | React + Vite + TS + Router，最轻量 |
| `react-admin` | **后台管理系统** | Ant Design Pro 或类似，含 Layout/权限/表格/表单组件 |
| `mobile-h5` | **移动端 H5** | Vue 3 + Vant，含移动端适配（viewport/rem） |
| `java-fullstack` | **全栈项目** | Spring Boot + React 前后端分离，含 API 调用示例 |
| `electron-app` | **桌面应用** | Electron + React/Vue IPC 通信示例 |
| `node-ts` | **Node 后端/API** | Express/Fastify + TS，含中间件/路由/错误处理示例 |
| `monorepo` | **多包仓库** | Turborepo + pnpm workspace，含包间依赖示例 |
| `chrome-ext` | **浏览器扩展** | Manifest V3，含 popup/content script/background 示例 |

**关键：每个模板只做一件事，不重复。**

#### 3. 模板内容的"干净度"标准

**必须有的**：
- ✅ 正确的项目结构（src/、public/、test/ 等）
- ✅ 已配好的工具链（ESLint、Prettier、TSConfig）
- ✅ 一个最小可运行的页面（Hello World 或 Layout 壳）
- ✅ README.md：项目结构说明 + 快速开始 + 架构决策说明

**绝对不能有的**：
- ❌ 硬编码业务数据（客户、订单、工单等）
- ❌ 特定行业的术语（华东渠道、门店、培训等）
- ❌ 超出模板定位的代码（SPA 模板里不要有后台表格）
- ❌ 脚手架自身的宣传（create-x-app badge、欢迎页）

#### 4. manifest.json 增强

```jsonc
{
  "key": "react-vite-ts",
  "name": "React + Vite + TypeScript",
  "description": "最轻量的 React SPA 起步模板",
  // 新增：模板定位标签
  "tags": ["spa", "frontend", "react", "vite"],
  // 新增：推荐场景
  "useCases": ["中小型前端应用", "单页应用", "管理后台前端"],
  // 新增：不推荐场景（帮用户避免选错）
  "notFor": ["移动端 H5（请用 mobile-h5）", "后台管理系统（请用 react-admin）"]
}
```

#### 5. 渐进式复杂度

每套模板的 `README.md` 里加一个"下一步"指引：

```markdown
## 下一步

拿到这个模板后，你可以：

1. **添加状态管理** → 推荐 Zustand / Jotai（轻量）或 Redux Toolkit（复杂场景）
2. **添加数据请求** → 推荐 TanStack Query + Axios
3. **添加 UI 组件库** → 推荐 Ant Design / shadcn/ui / MUI
4. **添加测试** → 推荐 Vitest + Testing Library
5. **添加部署** → 推荐 Dockerfile（已含示例）或 Vercel
```

---

### 执行优先级

| 优先级 | 任务 | 工作量 |
|--------|------|--------|
| **P0** | 清理所有模板中的硬编码业务数据，改为最小可运行代码 | 中 |
| **P0** | `react-vite-ts` 和 `react-admin` 定位拉开，去除重叠 | 中 |
| **P1** | 每套模板补充架构说明 README | 小 |
| **P1** | manifest.json 增加 tags / useCases / notFor | 小 |
| **P2** | minimal / example 分层（可选） | 大 |
| **P2** | 模板间增加对比表，帮用户选择 | 小 |

---

你想从哪个方向开始？我建议先做 **P0：清理硬编码业务数据 + 拉开 react-vite-ts 和 react-admin 的定位差异**，这是影响最大的改动。
